Context
Students and faculty at Columbia have no unified way to discover events across departments. Information is siloed on individual department websites or gated behind mailing lists. This project builds a public, no-login web hub that automatically scrapes event listings from multiple department sites and presents them in a searchable, filterable, calendar/map-enabled interface.
Team: 2–4 developers, ~12-week semester timeline
Language constraint: Python
Data strategy: Automated web scraping
Auth: None (fully public)

Core Data Models
Event
id, external_id (dedup hash), title, description, short_description,
start_datetime (UTC), end_datetime (nullable), all_day,
location_name, location_address, latitude, longitude,
source_url, image_url, registration_url, is_free, tags (JSON),
department_id (FK), category_id (FK, nullable),
created_at, updated_at, is_active, scrape_run_id (FK)

external_id = hash of (source_url + title) → prevents duplicate inserts on re-scrape
is_active = False when event disappears from source (never hard-delete)
All datetimes stored UTC; displayed as America/New_York in templates

id, name, slug, school, website_url, scraper_class,

    def get_event_urls(self) -> list[str]: ...

    @abstractmethod
    def parse_event(self, html: str, url: str) -> dict: ...

    def run(self) -> ScrapeResult:
        # Handles: rate limiting (1 req/sec), Playwright fallback,
        # deduplication, DB writes, ScrapeRun record, error capture
```

### Scheduling
- APScheduler runs inside the FastAPI process (single worker)
- One job per enabled department; interval = `department.scrape_interval_hours`
- `max_instances=1` per department — never double-runs
- `misfire_grace_time=600` — catches up within 10 min if server was restarted

### Error handling tiers
1. **Single URL 404/error**: log, skip, continue scraping rest of department
2. **Full department failure**: mark ScrapeRun "failed"; do NOT deactivate existing events yet; after 3 consecutive failures, emit warning
3. **Parse/schema error**: log bad record with source URL; skip that event; don't crash the scraper

## Frontend: HTMX Interaction Patterns

- **Filtering**: checkboxes/date picker → `hx-get="/events?category=X&start=Y"` → swaps `#event-list`
- **Search**: `hx-get="/search" hx-trigger="keyup changed delay:300ms"` → live results
- **Calendar navigation**: "Next month" → `hx-get="/calendar?month=2026-04"` → swaps grid
- **Map view**: server sets event coords as JSON in a `data-events` attribute; `map.js` reads it and initializes Leaflet
- **Save feature**: `saved_events.js` reads/writes event IDs to `localStorage`; shareable URL pattern `/saved?ids=1,2,3`
```

Critical Edge Cases & Technical Hurdles
Scraping

JS-rendered pages: use_playwright = True triggers Playwright; it's slow (~3–5s/page) — scrape only index pages with Playwright, not every detail page
Anti-scraping: Use polite rate limiting (1 req/sec) and a descriptive User-Agent header; back off on 429s rather than retrying aggressively
Inconsistent date strings: date_parser.py is the highest-risk module — build thorough unit tests for it in Week 2; wraps dateutil with custom fallbacks
Recurring events: "Every Tuesday through May" = multiple rows; build expand_recurring() in date_parser.py; fall back to single event + description note for complex rules
Event pages that 404 mid-scrape: Catch 404 explicitly, skip, log, continue

Data Quality

Cross-department duplicates: Same joint event posted by two departments → secondary dedup pass in deduplication.py matching on (title + start_datetime)
Location strings that don't geocode: Pre-populate location_overrides.json with 30 common Columbia building names; "Zoom"/"TBA" → latitude = NULL, excluded from map
Time zone ambiguity: Default all times without explicit TZ to America/New_York; flag explicit non-ET zones in a timezone_flag column

Architecture

Whoosh index persistence: Mount search_index/ on Railway persistent volume; add startup check to rebuild from DB if directory is empty
APScheduler + multiple workers: Never deploy with --workers > 1 (causes double-scraping); single worker is sufficient for this scale
Blocking I/O in async: Run all scraper HTTP calls in asyncio.run_in_executor() or switch to httpx (async-native) — decide at project start and be consistent
"Saved events" are browser-local: Communicate this clearly in the UI; offer shareable /saved?ids=1,2,3 URLs as a workaround
